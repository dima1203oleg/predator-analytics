Нижче наведено **повний перелік ключових компонентів** Predator Analytics v55.0 (Community Edition) з описом їхніх ролей, згрупований за функціональними категоріями. ## 1. ВСТУП

### 1.1. Що таке Predator

**Predator Analytics** — це відкрита аналітична платформа бізнес-розвідки, орієнтована на B2B та B2G ринки України. Система перетворює неструктурований хаос даних (митні декларації, податкові накладні, реєстри, медіа, Telegram) на точні, математично обґрунтовані **сигнали**, які дають користувачам інформаційну перевагу.

**Місія:** Дати користувачу здатність бачити приховане, передбачати майбутнє і приймати рішення на основі даних, а не інтуїції. Це інструмент для контролю над невизначеністю.

**Ключові можливості:**
- **Ринкова розвідка:** моніторинг конкурентів у реальному часі.
- **Кредитний рентген:** оцінка реального обороту та ризиків контрагентів.
- **Макроекономічна картина:** агреговані показники, структурні зрушення, прогнози.
- **Детектор схем:** виявлення аномалій, фантомних потоків, прихованих зв'язків.
- **Карта впливу:** графічне представлення концентрації капіталу та впливу.
- **Машина передбачення:** ймовірнісні сценарії розвитку подій.

### 1.2. Ключові принципи проєктування

1.  **Signal over Data:** Користувач отримує не доступ до даних, а сигнали (інсайти, попередження, прогнози).
2.  **Layered Analysis:** Кожен шар аналітики (Behavioral, Institutional, Influence, Structural, Predictive) має чітку математичну формалізацію.
3.  **Entity-Centric:** Всі дані зшиваються навколо унікального ідентифікатора UEID.
4.  **Probabilistic Thinking:** Всі висновки мають ймовірнісний характер з Confidence Score.
5.  **Explainability by Law:** Кожен сигнал супроводжується SHAP-поясненням та джерелами.
6.  **Reactive Architecture:** Система будується навколо подій (event-driven).
7.  **Scale from Day One:** Архітектура проєктується під горизонтальне масштабування.
8.  **Security Embedded:** Безпека вбудована в кожен шар (Zero Trust, mTLS, Vault).
9.  **Ethical Boundaries:** Система не надає інструментів для маніпуляцій чи персональних атак.

### 1.3. Мета документа

Цей документ є **єдиним канонічним джерелом істини** для розробки, розгортання, тестування та експлуатації платформи Predator v55.0. Він призначений для:
- **Інженерної команди** — як вичерпне керівництво до дії.
- **Product-менеджменту** — як фіксація вимог.
- **Керівництва та інвесторів** — як демонстрація готовності до реалізації.
- **Державних замовників** — як підтвердження відповідності стандартам.

## 2.5. PREDATOR: PREDICTIVE ECONOMIC INTELLIGENCE PLATFORM
*(The Paradigm Shift)*

Представляє новий клас продукту: **Predictive Economic Intelligence Platform**. Це не просто OSINT або ретроспективний аналіз. **Predator — це машина прогнозування бізнесу.**
Завдяки злиттю 8 років історичних даних (Митниця + Податкова + OSINT + Ринок) система здатна:
1. Знаходити складні патерни імпорту / продажів / попиту.
2. Будувати ймовірнісні моделі прогнозування бізнес-поведінки цілих ринків.
3. Давати бізнесу точні рекомендації.

**Архітектура прогнозування базується на 4 двигунах AI:**
1. **Pattern Engine:** Пошук сезонності, циклів та залежностей у історичних даних (8 років).
2. **Market Forecast Engine:** Прогнозування ринку використовуючи XGBoost, LSTM, Prophet, ARIMA для передбачення попиту та інвентарю.
3. **Behavior Engine:** Прогнозування поведінки компаній (коли почне імпорт, коли змінить товар, коли ризик банкрутства).
4. **Recommendation Engine:** Найвищий рівень цінності (Що імпортувати / Коли / Звідки / Скільки).

**5 рівнів прогнозування платформи:**
1. *Прогноз попиту* (зростання попиту на певну групу в певному регіоні).
2. *Прогноз імпорту* (хто і коли почне завозити товар).
3. *Прогноз цін* (ма?. **Market Forecast Engine:** Прогнозуванняни через зміну логістики).
4. *Прогноз конкурентів* (хто новий зайде на ринок).
5. *Прогноз ризиків* (санкції, суди, банкрутства під лінзою майбутнього).

### 3.14. Confidence Score

```
CS = w₁·DataCompleteness + w₂·ModelStability + w₃·HistoricalAccuracy
   + w₄·IndexVariance + w₅·(1 - DriftScore)

де:
  w₁ = 0.25  (повнота даних: % заповнених полів)
  w₂ = 0.20  (стабільність моделі: std останніх N прогнозів)
  w₃ = 0.25  (історична точність: accuracy на validation set)
  w₄ = 0.15  (варіативність індексу: 1 - CV)
  w₅ = 0.15  (зворотний дрейф: 1 - drift_score)
```

**Реалізація:** `app/core/confidence.py`

## 6. МАТЕМАТИЧНІ ІНДЕКСИ ТА ФОРМУЛИ

### 6.1. Behavioral Volatility Index (BVI)

**Файл:** `app/indices/bvi.py`

```
BVI = σ(ΔV) / μ(V) × 100

де:
  σ(ΔV) = стандартне відхилення змін обсягу (volume) за період
  μ(V)  = середній обсяг за період
```

**Інтерпретація:** Чим вищий BVI, тим нестабільніша поведінка суб'єкта. BVI > 60 — критичний сигнал.

### 6.2. Adaptation Speed Score (ASS)

**Файл:** `app/indices/ass.py`

```
ASS = Δ_behavior / Δ_regulation × k

де:
  Δ_behavior   = зміна патерну імпорту після регуляторної зміни
  Δ_regulation = час з моменту введення нового правила
  k             = нормалізуючий коефіцієнт (sector-specific)
```

**Інтерпретація:** Висока швидкість адаптації (ASS > 0.8) може свідчити про інсайдерську інформацію або заздалегідь підготовлені схеми.

### 6.3. Collapse Probability (CP)

**Файл:** `app/indices/cp.py`

```
CP = σ(BVI, financial_stress, debt_ratio, legal_events)

де:
  σ = сигмоїда (logistic function)
  BVI             = Behavioral Volatility Index
  financial_stress = коефіцієнт фінансового навантаження
  debt_ratio       = рівень боргового навантаження
  legal_events     = кількість судових справ за останні 12 місяців
```

**Інтерпретація:** CP > 0.7 — висока ймовірність банкрутства протягом 6-12 місяців.

### 6.4. Administrative Asymmetry Index (AAI)

**Файл:** `app/indices/aai.py`

```
AAI = |T_actual - T_expected| / T_expected × 100

де:
  T_actual   = фактичний час розмитнення для суб'єкта
  T_expected = очікуваний час (median для регіону/товару)
```

**Інтерпретація:** AAI > 50 — значна аномалія, потенційне порушення або корупція.

### 6.5. Post Loyalty Score (PLS)

**Файл:** `app/indices/pls.py`

```
PLS = count(declarations_at_post) / total_declarations × 100

де:
  count(declarations_at_post) = кількість декларацій на конкретному посту
  total_declarations            = загальна кількість декларацій суб'єкта
```

**Інтерпретація:** PLS > 80 — критична прив'язка до одного посту (потенційна схема).

### 6.6. Influence Mass (IM)

**Файл:** `app/indices/im.py`

```
IM = α × PageRank + β × BetweennessCentrality + γ × EigenvectorCentrality

де:
  α = 0.4 (вага PageRank)
  β = 0.3 (вага BetweennessCentrality)
  γ = 0.3 (вага EigenvectorCentrality)
```

**Інтерпретація:** IM > 0.7 — ключовий гравець у мережі.

### 6.7. Hidden Concentration Index (HCI)

**Файл:** `app/indices/hci.py`

```
HCI = 1 - H(market_shares) / H_max

де:
  H(market_shares) = ентропія Шеннона розподілу ринкових часток
  H_max             = max ентропія (рівний розподіл)
```

**Інтерпретація:** HCI > 0.6 — висока прихована концентрація ринку (монополізація).

### 6.8. Missing Chain Index (MCI)

**Файл:** `app/indices/mci.py`

```
MCI = |Import - (DomesticSales + Export + ΔInventory)| / Import × 100

де:
  Import         = сума імпорту (митні декларації)
  DomesticSales  = продажі на внутрішньому ринку
  Export         = експорт
  ΔInventory     = зміна запасів
```

**Інтерпретація:** MCI > 30 — критична невідповідність (можливий тіньовий оборот).

### 6.9. Phantom Flow Index (PFI)

**Файл:** `app/indices/pfi.py`

```
PFI = Σ(unmatched_flows) / Σ(total_flows) × 100

де:
  unmatched_flows = транзакції без відповідного контрагента
  total_flows     = загальна кількість транзакцій
```

**Інтерпретація:** PFI > 20 — значна кількість "зниклих" потоків.

### 6.10. Composite Economic Risk Score (CERS)

**Файл:** `app/engines/cers.py`

```
CERS = Σ(wᵢ × normalize(Indexᵢ))

де:
  wᵢ = адаптивна вага (sector/region-specific)
  normalize = Z-score normalization + PCA decorrelation
  Indexᵢ ∈ {BVI, ASS, CP, AAI, PLS, IM, HCI, MCI, PFI}

Ваги за замовчуванням (загальний профіль):
  w_BVI = 0.12
  w_ASS = 0.08
```

## 7. ВЕБ-ІНТЕРФЕЙС (v4.0 STRUCTURE)

### 7.1. Загальна концепція та Навігація
Інтерфейс побудований навколо **4 головних режимів**, як "GPS для бізнесу". Усі 180+ мікрокомпонентів згорнуті в логічну екосистему для спрощеного користування.

**Глобальні елементи інтерфейсу (видимі завжди):**
- **Top Bar:** Логотип | Універсальний Object Search | AI Copilot (кнопка) | Центр Сповіщень (Real-time anomalies) | Профіль/Підписка.
- **Sidebar (Ліве меню):** 4 Великі режими: "Ринок", "Прогноз", "Можливості", "Ризики". Швидкі лінки: Economic Graph Explorer, AI Agents Feed, Due Diligence Quick Check, Market Simulator.
- **AI Copilot Panel:** Чат на природній мові (RAG) з контекстом системи + історією запитів.
- **Notification Center:** Сигнали про нових конкурентів, аномалії цін, нові AI-рекомендації.

---

### 7.2. Головні Режими (The Core Pillars)

#### РЕЖИМ 1: РИНОК (Market) – COMP-156
*Мета: зрозуміти поточну ситуацію ринку прямо зараз.*
- **Дашборд ринку:** KPI (загальний імпорт, Топ-10 товарів/країн, частка ринку по компаніях, динаміка за 12 міс).
- **Радар конкурентів (Competitor Radar):** Real-time стрічка імпортних транзакцій конкурентів.
- **Карта торгових потоків:** Інтерактивна MapLibre GL мапа імпорту/експорту з фільтрами.
- **Тренд-сканер (Trend Scanner):** Виявлення нових/зростаючих категорій, нових імпортерів та постачальників за останні 30 днів.

#### РЕЖИМ 2: ПРОГНОЗ (Forecast) – COMP-157
*Мета: побачити майбутнє на 1–12 місяців.*
- **Прогноз попиту (Demand Prediction):** Графіки та довірчі інтервали по категоріях товарів.
- **Прогноз цін (Price Forecast):** Прогнозування закупівельної та митної вартості.
- **Прогноз імпорту конкурентів (Import Forecast):** Обсяги, що конкуренти планують імпортувати.
- **Аналіз насичення ринку:** Оцінка рівня насичення та визначення лімітів.
- **Market Simulator (Що-якщо):** Введення кастомних макро/мікро сценаріїв та отримання прогнозу маржі чи реакції ринку.

#### РЕЖИМ 3: МОЖЛИВОСТІ (Opportunities) – COMP-158
*Мета: знайти найприбутковіші ніші.*
- **Товар & Країна тижня:** Авто-рекомендовані товари з маржею 30-70% та найвигідніші джерела.
- **Recommendation Engine Report:** Таблиця точних рекомендацій (Що / Звідки / Коли / Скільки / Очікувана Маржа / Confidence Score).
- **Market Opportunity Engine:** Картки зростаючих "гарячих ніш" з високим попитом та низькою конкуренцією.

#### РЕЖИМ 4: РИЗИКИ (Risks) – COMP-159
*Мета: збереження капіталу.*
- **Стрічка аномалій & Сигналів:** Цінові стрибки, різка поява нових гравців.
- **Due Diligence Quick Check:** Повне досьє компаній (ЄДРПОУ, бенефіціари, податки, CERS Score).
- **Санкційні Списки:** Real-time чекінг контрагентів.
- **Прогноз макроризиків:** Вплив курсу, логістики чи інфляції на портфель клієнта.

### 8.2. Канонічні пайплайни

```
Upload → Validate → Classify → Parse → Normalize → Enrich → Index → Signal
  │         │          │         │         │          │        │        │
  ▼         ▼          ▼         ▼         ▼          ▼        ▼        ▼
 MinIO   Schema    ML-type    Extract   UEID     Geocode   OpenSearch  Kafka
         check     detect     fields    assign   NER       Qdrant    Signal Bus
                                                           Neo4j
```

**Крок 1 — Upload:** Файл зберігається в MinIO (COMP‑106). Генерується `job_id`.
**Крок 2 — Validate:** Перевірка формату, розміру, цілісності.
**Крок 3 — Classify:** ML-класифікація типу документа.
**Крок 4 — Parse:** Витягнення структурованих полів.
**Крок 5 — Normalize:** Нормалізація назв, дат, адрес, присвоєння UEID.
**Крок 6 — Enrich:** Збагачення (геокодування, NER, sentiment).
**Крок 7 — Index:** Індексація в OpenSearch (повнотекст), Qdrant (ембедінги), Neo4j (зв'язки).
**Крок 8 — Signal:** Публікація події `data.ingested` у Kafka (Signal Bus).

### 8.3. Прогрес у реальному часі (SSE)

```
GET /api/v2/jobs/{job_id}/progress
Content-Type: text/event-stream

data: {"step": "parse", "progress": 45, "message": "Витягнення полів..."}
data: {"step": "enrich", "progress": 72, "message": "NER аналіз..."}
data: {"step": "complete", "progress": 100, "message": "Готово"}
```

### 8.4. Job Lifecycle Management

| Статус | Опис |
|--------|------|
| `pending` | Завдання створено, очікує обробки |
| `processing` | Активна обробка |
| `enriching` | Збагачення даних |
| `indexing` | Індексація |
| `completed` | Успішно завершено |
| `failed` | Помилка (з деталями) |
| `cancelled` | Скасовано користувачем |

## 13. БЕЗПЕКА, АУДИТ ТА КОНТРОЛЬ ПОМИЛОК

### 13.1. Zero Trust Architecture (Single Node)

- **Принцип:** "Never trust, always verify."
- **mTLS:** Забезпечується через базовий Cilium (без важких Service Mesh на кшталт Istio Ambient, що недоцільні для одного вузла).
- **JWT:** Кожен запит до API авторизується через Keycloak-виданий JWT.
- **Network Policies:** Кожен под має whitelist дозволених з'єднань (Cilium Network Policies).
- **Vault:** Усі секрети зберігаються у HashiCorp Vault.

### 13.2. Confidence Score

Кожен AI-генерований сигнал має обов'язковий `confidence_score` (0.0 – 1.0):

| Діапазон | Маркування | Дія |
|----------|------------|-----|
| 0.8 – 1.0 | Високий | Автоматичне сповіщення |
| 0.5 – 0.79 | Середній | Потребує перевірки |
| < 0.5 | Низький | Лише для інформації, не для дій |

### 13.3. False Positive Calibration

- **Feedback Loop:** Аналітики позначають помилкові сигнали через UI.
- **Ретренування:** Модель перетренується щомісяця з урахуванням фідбеку.
- **Target FPR:** < 5% для Critical сигналів, < 15% для Watchlist.
- **Метрика:** Precision@Recall=0.9 для CERS > 80.

### 13.4. Manual Override

- Будь-який автоматичний сигнал може бути перевизначений уповноваженим аналітиком.
- Override записується в `decision_artifacts` з причиною та підписом.
- Override не змінює оригінальний сигнал, а створює новий артефакт.

### 13.5. Версіонування моделей (MLflow)

- **Кожна модель** має унікальний `model_fingerprint` (hash ваг + гіперпараметрів).
- **Model Registry** (MLflow) зберігає всі версії з тегами: staging, production, archived.
- **Rollback:** Можливість повернутися до попередньої версії за < 5 хвилин.

### 13.6. Аудит рішень (Decision Artifacts)

**Таблиця:** `decision_artifacts` (PostgreSQL, WORM)

```sql
CREATE TABLE decision_artifacts (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    decision_id     TEXT NOT NULL UNIQUE,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    tenant_id       TEXT NOT NULL,
    trace_id        TEXT NOT NULL,
    decision_type   TEXT NOT NULL,
    input_fingerprint  TEXT NOT NULL,
    model_fingerprint  TEXT NOT NULL,
    output_fingerprint TEXT NOT NULL,
    confidence_score   FLOAT NOT NULL CHECK (confidence_score BETWEEN 0 AND 1),
    input_snapshot     JSONB NOT NULL,
    output_snapshot    JSONB NOT NULL,
    explanation        JSONB,
    metadata           JSONB DEFAULT '{}'
);

-- WORM: забороняємо UPDATE та DELETE
CREATE TRIGGER prevent_update_decision_artifacts
    BEFORE UPDATE ON decision_artifacts FOR EACH ROW
    EXECUTE FUNCTION raise_immutable_error();

CREATE TRIGGER prevent_delete_decision_artifacts
    BEFORE DELETE ON decision_artifacts FOR EACH ROW
    EXECUTE FUNCTION raise_immutable_error();
```

**Retention:** 7+ років (legal compliance).

### 13.7. Ethical Audit

- **Щоквартальний аудит:** Незалежна перевірка використання AI-рішень.
- **Bias Detection:** Автоматична перевірка на дискримінаційні патерни.
- **Канал для викривачів:** Анонімний звіт через `/ethics/report`.
- **Публічний звіт:** Щорічна публікація статистики AI-рішень.

### 13.8. SIO & Manual Gate (Запобігання самодеструкції)

**Self-Improvement Orchestrator (SIO)** на Single Node без staging кластера є критичним ризиком. 
Тому впроваджено обов'язковий **Manual Gate**:
- SIO створює Pull Request для зміни схем, конфігів чи порогів (Auto PR).
- SIO **не має права** робити автоматичний merge або auto-sync в ArgoCD.
- Будь-яка зміна потребує Approve від людини (L3 Authority).
- У разі помилки є чітка Rollback Policy через Git reverse commit.
