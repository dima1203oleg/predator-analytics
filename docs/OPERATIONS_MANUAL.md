# PREDATOR Analytics v56.5-ELITE — Інструкція з експлуатації (OPERATIONS_MANUAL)

Цей документ призначений для SRE-інженерів та адміністраторів системи. Він описує процедури підтримки стабільності та безпеки платформи.

---

## 1. Архітектурний огляд
PREDATOR функціонує як сукупність мікросервісів у K3s кластері, об'єднаних шиною подій Kafka.
- **Frontend**: React (порт 3030 / 80)
- **Core API**: FastAPI (порт 8000)
- **Pulse Engine**: ML-прогнозування
- **Antigravity**: AGI-оркестратор

## 2. Моніторинг та Sentinel
Система використовує **Sentinel Service** для глибокої перевірки здоров'я.

### Ендпоїнти перевірки:
- `/health`: Базова перевірка аптайму.
- `/ready`: Перевірка всього AI-стеку (Redis, Kafka, Neo4j, LiteLLM).

**Статуси Sentinel:**
- `ready`: Всі системи функціонують штатно.
- `degraded`: Один або декілька критичних компонентів (Kafka/Neo4j) недоступні.

## 3. Усунення несправностей (Troubleshooting)

### 3.1. Стан DEGRADED
Якщо Sentinel повертає 503:
1. Перевірте стан чарту через ArgoCD.
2. Виконайте `kubectl get pods -n predator` для пошуку Restarting подів.
3. Перевірте логи Sentinel: `kubectl logs -l app=core-api | grep SENTINEL`.

### 3.2. Chaos Engineering (Фаза 7)
Якщо система поводиться нестабільно через активні хаос-експерименти:
1. Перейдіть в **Admin Hub -> Chaos Ops**.
2. Натисніть **ABORT ALL EXPERIMENTS**.
3. Усі експерименти мають TTL 15 хв і вимкнуться автоматично, але ручна зупинка є пріоритетною.

## 4. Безпека та mTLS (Фаза 8)
Усі міжсервісні запити захищені mTLS.
- **Ротація сертифікатів**: Виконується через `cert-manager` у Kubernetes.
- **Zero-Trust**: Якщо сервіс не має валідного представлення в `mtls.py`, запит буде відхилено з помилкою 403.

## 5. Прогностичне ядро (Фаза 9)
Прогнози попиту залежать від стабільності бази даних.
- Якщо `confidence_score` у прогнозах низький (< 0.5), перевірте затримку Neo4j та наявність активних хаос-ін'єкцій.
