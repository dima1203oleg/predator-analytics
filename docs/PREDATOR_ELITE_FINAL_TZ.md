## 1. Концептуальне введення та стратегічна місія

PREDATOR Analytics v56.5-ELITE — це суверенна OSINT-платформа для митної аналітики та виявлення ризиків, що працює в режимі автономної зброї фінансової розвідки. Система поєднує графові бази даних, AI-агенти та потокову обробку подій для миттєвого виявлення аномалій та прихованих схем.

**Стратегічна місія:** Перетворити PREDATOR Analytics на систему розвідувального класу, здатну працювати автономно, виявляти загрози в реальному часі та забезпечувати юридичну чистоту розслідувань.

## 2. Онтологія графової бази (Neo4j)

### 2.1. Ключові сутності (Nodes)

- **Company** — Компанія/Юридична особа/ФОП (edrpou, name, status, registration_date, is_fop)
- **Person** — Фізична особа (inn, full_name, citizenship, is_pep, is_sanctioned)
- **Declaration** — Митна декларація (decl_id, date, type, total_invoice_value, currency)
- **Product** — Товар/УКТЗЕД (hs_code, description, category)
- **CustomsPost** — Митний пост/КПП (post_code, name, region, location)
- **Broker** — Митний брокер (license_id, name)
- **Address** — Локація/Юридична адреса (address_hash, full_address)
- **Country** — Країна (iso_code, name, is_offshore, is_sanctioned)

### 2.2. Типи зв'язків (Relationships)

**Бізнес-зв'язки:**
- `(Person)-[:DIRECTS {appointment_date}]->(Company)` — Особа керує компанією
- `(Person)-[:OWNS {share_percent}]->(Company)` — Особа володіє часткою компанії
- `(Company)-[:REGISTERED_AT]->(Address)` — Компанія зареєстрована за адресою

**Митні операції:**
- `(Company)-[:FILED]->(Declaration)` — Компанія подала декларацію
- `(Broker)-[:PROCESSED]->(Declaration)` — Брокер оформив декларацію
- `(Declaration)-[:CLEARED_AT]->(CustomsPost)` — Декларація пройшла через митний пост
- `(Declaration)-[:CONTAINS {net_weight, gross_weight, invoice_value, quantity}]->(Product)` — Декларація містить товар
- `(Declaration)-[:ORIGINATES_FROM]->(Country)` — Товар походить з країни
- `(Declaration)-[:DISPATCHED_FROM]->(Country)` — Товар відправлено з країни

### 2.3. Архітектурні принципи

1. **Аналітична вага на ребрах** — Суми, обсяги, вага зберігаються на зв'язках для швидкого проходження графом без підняття важких таблиць.
2. **Унікальність через constraints** — Жорсткі обмеження унікальності запобігають дублюванню даних.
3. **Індекси для аналітики** — Індекси оптимізовані для типових аналітичних запитів (дати, суми, коди).
4. **Гнучкість для 100+ датасетів** — Онтологія підтримує всі типи аналітичних патернів: бізнес-зв'язки, митні операції, географічні залежності.

**Детальна документація:** `docs/neo4j_ontology.md`

## 3. Нейро-оркестратор (LLM Agentic Reasoning Layer)

### 3.1. ReAct (Reasoning + Acting) Loop

Агент працює за циклом: Отримати запит → Створити Cypher-запит → Виконати → Побачити помилку або порожню відповідь → Самостійно виправити запит → Видати результат.

**Переваги:**
- **Автономна самокорекція** — Агент самостійно виправляє помилки в запитах
- **Прозорість мислення** — Кожен крок логується для аудиту
- **Адаптивність** — Агент адаптується під різні типи запитів

### 3.2. Tool Calling API (Функціональні виклики)

Набір жорстко зашитих функцій (Tools), які агент може викликати для виконання специфічних завдань:

- **search_graph_anomaly** — Пошук аномалій в графі за заданим патерном
- **get_entity_details** — Отримання детальної інформації про сутність (компанію, особу, декларацію)
- **analyze_risk_score** — Аналіз ризик-скорингу компанії або особи

**Переваги:**
- **Модульність** — Кожен інструмент може бути розроблений та тестований ізольовано
- **Безпека** — Інструменти обмежені конкретними операціями
- **Масштабованість** — Легко додавати нові інструменти

### 3.3. Vector Semantic Router

Перед тим як генерувати запит, система через Qdrant шукає в базі "схожі минулі розслідування", щоб зрозуміти контекст і використати вже перевірені алгоритми пошуку.

**Переваги:**
- **Контекстна обізнаність** — Агент використовує досвід попередніх розслідувань
- **Прискорення** — Схожі запити обробляються швидше
- **Навчання** — Система покращується з кожним розслідуванням

### 3.4. Інтеграція з DeepSeek R1

Нейро-оркестратор використовує DeepSeek R1 як основну LLM модель для генерації запитів та аналізу результатів.

**Детальна документація:** `docs/neuro_orchestrator.md`

## 4. Система потокового перехоплення (Event-Driven Pipeline)

### 4.1. Шина даних (Redpanda / Kafka)

Усі нові дані від конекторів (YouControl, Opendatabot, Митниця) потрапляють не напряму в базу, а в швидкісну чергу повідомлень (Message Broker) — Redpanda (бо вона легша за Kafka і написана на C++, що ідеально для вашого сервера).

**Топіки (Topics):**
- `stream.customs.declarations` — Митні декларації
- `stream.registry.fop_changes` — Зміни реєстру ФОП
- `stream.prozorro.tenders` — Тендери Prozorro
- `stream.court.decisions` — Судові рішення
- `stream.sanctions.updates` — Оновлення санкційних списків

### 4.2. Complex Event Processing (CEP-рушій)

Ми використовуємо потоковий обробник (наприклад, Faust для Python або Apache Flink), який "слухає" ці топіки. Як тільки надходить нова декларація, CEP-рушій миттєво перевіряє її на відповідність 100 закладеним датасетам аномалій.

### 4.3. Механізм Алертів (Alerting & Triage)

Якщо CEP-рушій знаходить збіг (Pattern Match):
1. Дані записуються в TimescaleDB (база часових рядів) як Incident Event.
2. Графана (через Alertmanager) генерує візуальний сигнал у "Sovereign Command Center".
3. Нейро-оркестратор (Агент) автоматично отримує тригер, збирає досьє на всіх пов'язаних осіб і формує попередній звіт ще до того, як аналітик відкриє вкладку.

**Детальна документація:** `docs/event_streaming.md`

## 5. Підсистема злиття сутностей (Entity Resolution Pipeline)

### 5.1. Трирівнева архітектура дедуплікації

Цей процес запускається на етапі ETL-конвеєра (до запису в Neo4j):

1. **Детерміноване злиття (Exact Match)** — Жорстке злиття за унікальними ідентифікаторами (ЄДРПОУ, ІПН, номер ліцензії).
2. **Ймовірнісне злиття (Fuzzy Matching)** — Використання алгоритмів Jaro-Winkler або Levenshtein distance для назв та адрес.
3. **AI-Дизамбігуація (LLM Entity Resolution)** — Для найскладніших випадків. Якщо є два "Іванови І.І." без ІПН, але вони перетинаються в історії володіння суміжними компаніями, у гру вступає локальна модель (наприклад, GLM-5.1).

### 5.2. Стратегія збереження оригіналів (SAME_AS Edges)

В розвідці не можна фізично видаляти сирі дані. Замість того, щоб перезаписувати вузли, система залишає оригінальні записи, але об'єднує їх спеціальним ребром:

```
(Person_Record_1)-[:SAME_AS {confidence: 0.98, method: "AI"}]->(Person_Master_Node)
(Person_Record_2)-[:SAME_AS {confidence: 0.95, method: "FUZZY"}]->(Person_Master_Node)
```

Аналітичні запити завжди йдуть через `Master_Node`, але аналітик в UI може "провалитися" до оригінального брудного запису.

**Детальна документація:** `docs/entity_resolution.md`

## 6. Архітектура нульової довіри та WORM-аудит (Zero-Trust & Compartmentalization)

### 6.1. Рольова ізоляція даних (RLS + Graph Compartmentalization)

Інтеграція Keycloak із базами даних повинна бути наскрізною:

- **PostgreSQL RLS (Row-Level Security)** — Кожен запит до TimescaleDB автоматично обгортається Tenant ID з JWT-токена Keycloak.
- **Neo4j Node-Level Security** — Налаштування кастомних ролей у графовій базі. Якщо аналітик має доступ лише до "Митного поста А", система відфільтрує граф так, що ребра, пов'язані з "Митним постом Б", будуть для нього "прозорими".

### 6.2. Криптографічна незмінність доказів (WORM - Write Once Read Many)

Для забезпечення юридичної та оперативної чистоти розслідувань впроваджується протокол HR-16:

**Decision Ledgers (Журнали рішений):**
Кожен висновок, згенерований AI-агентом, та кожна знайдена аномалія записуються у спеціальну таблицю з криптографічним підписом (hash-chain). Це гарантує, що якщо "Pattern Engine" знайшов незаконну схему, жоден адміністратор бази даних не зможе "підчистити" цей запис заднім числом без руйнування ланцюга хешів.

### 6.3. Мережевий периметр (Cilium & Vault)

- **Default-Deny L3/L4** — Cilium блокує будь-який трафік між контейнерами. Docker-контейнер з UI не має права стукати до бази Neo4j — тільки до API Backend.
- **Vault Secret Rotation** — Паролі до баз (Postgres, ClickHouse, Neo4j) не існують у статичному вигляді. HashiCorp Vault генерує динамічні облікові дані, які "живуть" лише 15 хвилин.

**Детальна документація:** `docs/zero_trust_security.md`

## 8. Мережева автономія та Резильєнтність (NVIDIA_ZROK)

### 8.1. ZROK Tunneling (Bypass & Proxy)

**P2P Оверлей:**
ZROK (на базі OpenZiti) створює зашифрований peer-to-peer тунель. Це дозволяє отримувати доступ до localhost:3030 (UI) та localhost:8000 (API) з будь-якої точки світу без відкриття портів на роутері.

**Dark Network:**
Система не "відсвічує" в інтернеті (не сканується через Shodan), оскільки не має публічних відкритих портів, окрім P2P-з'єднання, яке вимагає криптографічної аутентифікації.

### 8.2. Agentic Sync (Синхронізація Знань)

Якщо розгортається кілька "бойових" нод системи (наприклад, у різних підрозділах), векторні бази (Qdrant) та журнали рішень обмінюються даними через ZROK.

Якщо Нода А виявила новий патерн поведінки (наприклад, нову схему ухилення на митниці), знання про цей патерн автоматично передаються в GLM-5.1 на Ноді Б.

**Детальна документація:** `docs/network_autonomy.md`

## 9. Управління ресурсами (SM Resource Contract)

### 9.1. GPU Memory Manager (Квантування та Offloading)

**LLM Budget $0:**
Моделі (DeepSeek R1 або GLM-5.1) не можуть працювати у FP16 на 8GB VRAM. У ТЗ фіксується використання формату GGUF (int4/int8 quantization).

Модель завантажується через Ollama. Шари моделі, які не поміщаються у 8GB VRAM відеокарти, автоматично скидаються (offloading) у системні 64GB RAM.

### 9.2. Бази даних та Swap/Cache

- **PostgreSQL (TimescaleDB) та ClickHouse** — Налаштовуються з жорстко лімітованими пулами пам'яті (наприклад, ClickHouse max_server_memory_usage = 8GB, PostgreSQL shared_buffers = 4GB).
- **Redis** — Працює з політикою витіснення allkeys-lru і жорстким лімітом у 2GB.

### 9.3. Життєвий цикл розгортання (deploy_sm.sh)

Єдиний баш-скрипт відповідає за холодний старт:
1. Перевірка наявності NVIDIA Drivers та Container Toolkit.
2. Створення правильної ієрархії тек для Volumes.
3. chmod / chown — налаштування прав доступу (security hardening).
4. Запуск Trivy — сканування Docker-образів на вразливості перед їхнім підняттям.
5. Запуск docker-compose up -d.

**Детальна документація:** `docs/resource_management.md`
