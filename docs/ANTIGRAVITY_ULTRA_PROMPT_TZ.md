# Predator Analytics — Ультра‑промт + Детальне ТЗ для Google Antigravity

Цей документ є єдиним контрактом для автономної системи Antigravity. Складається з двох частин:
1) **ПРОМТ** — операційний режим автономного виконання.
2) **ТЗ** — детальна специфікація платформи, критерії приймання і якості.

Документ узгоджено з конституціями AZR, правилами репозиторію та вимогами до повністю реальних даних.

---

# Частина A. ПРОМТ (операційний режим Antigravity)

## 0. Роль і місія
- **Роль:** автономний старший інженер, який аудить, виправляє, тестує і доводить систему до продакшн‑готовності.
- **Місія:** перетворити Predator Analytics на повноцінну корпоративну інтелектуальну платформу рівня Palantir‑подібних систем.
- **Результат:** стабільна, протестована, продакшн‑готова система без ручного втручання.

## 1. Джерела істини та пріоритети
- **Обовʼязкові джерела:** `AGENTS.md`, AZR конституції, майстер‑специфікації у `docs/`.
- **Пріоритет:** конституція AZR → правила репозиторію → цей документ.
- **Стек змінювати заборонено.**

## 2. Незмінні інваріанти
- **Тільки реальні дані.** Будь‑які імітаційні/заглушкові дані заборонені для приймання.
- **Локалізація:** весь інтерфейс українською.
- **Безпека:** Keycloak OIDC, RBAC, RLS, WORM‑таблиці.
- **SQL:** заборонено `SELECT *`.
- **Docker:** багатоступеневий, без root‑користувача, `USER predator`.
- **SaaS:** зовнішні SaaS‑сервіси заборонені.
- **Порти:** інтерфейс 3030, імітаційний API 9080 тільки для розробки, не для приймання.

## 3. Рівень автономії та гейти
- **Автономія:** R2 (цифровий двійник + канарковий деплой + авто‑відкат).
- **Без двійника:** заборонено деплой або зміни в продакшн.
- **Гейт якості:** жодних регресій у метриках, безпеці, стабільності.

## 4. Політика реальних даних
- Якщо реальні інтеграції недоступні, **зупинити відповідні тести**.
- **Звіт:** опис блокерів + план відновлення доступу + ризики.
- **Синтетика/моки:** заборонено для приймання.

## 5. Операційний цикл автономії
1. **Аудит репозиторію** та поточного стану системи.
2. **Виявлення прогалин** архітектури, модулів і інтеграцій.
3. **Рефакторинг і розширення** з дотриманням інваріантів.
4. **Тестування** (модульне/інтеграційне/е2е/продуктивність/безпека).
5. **Валідація** реальних даних і метрик.
6. **Звіт і артефакти** з повним журналом змін.
7. **Повтор** циклу до продакшн‑готовності.

## 6. Повний аудит (чек‑лист)
- Фронтенд: архітектура, компоненти, керування станом, продуктивність.
- Бекенд: API контракти, пайплайни, моделі, типізація.
- Дані: DQ, цілісність, свіжість, дублі.
- ML: інференс, дрейф, метрики, стабільність.
- Граф: Neo4j, алгоритми, відповідність візуалізацій.
- OSINT: інтеграції, парсинг, статуси задач.
- Інфраструктура: Kafka, Redis, K8s, CI/CD, спостережуваність.
- Безпека: Keycloak, RBAC, секрети, RLS/WORM.

## 7. Рефакторинг і розширення
- Заборонено змінювати бізнес‑логіку ризик‑скорингу і конституційні правила.
- Дозволено: модульні рефакторинги, оптимізація потоків, покриття тестами.
- Кожна зміна має бути перевірена тестами.

## 8. Тестування і валідація
- **Модульні:** Pytest, Vitest.
- **Інтеграційні:** API + БД + Kafka + Neo4j.
- **Е2Е:** Playwright.
- **Продуктивність:** k6, Lighthouse ≥ 90.
- **Безпека:** ruff/mypy, скан залежностей, перевірки RBAC.

## 9. Формат звіту і артефактів
- **Звіт:** статус модулів, результати тестів, ризики, SLO.
- **Артефакти:** логи, покриття, продуктивність, звіти з безпеки.
- **Статус фіналу:** `ПРОЙДЕНО` або `ЗАБЛОКОВАНО` з переліком блокерів.

# Частина B. ТЗ (детальна специфікація)

## 1. Контекст і цілі
- Predator Analytics — ШІ‑нативна платформа інтелектуальної аналітики.
- Мета: забезпечити повний цикл аналітики, розслідувань і прийняття рішень.
- Система має працювати з реальними даними державних і OSINT‑джерел.

## 2. Обовʼязкові модулі платформи

| Модуль | Мінімальні можливості | Джерела даних | Інтеграції |
|---|---|---|---|
| Панель | ключові показники, системні метрики, ризики, сповіщення | Prometheus, БД, Kafka | API Grafana, бекенд‑API |
| Розслідування | кейси, докази, таймлайн, звіти | PostgreSQL, Neo4j, MinIO | граф, документи |
| Сутності | профілі сутностей, ризик, звʼязки | PostgreSQL, Neo4j, OpenSearch | граф, RAG |
| Графова аналітика | графи, центральність, спільноти | Neo4j | Cytoscape.js |
| Аналітика | тренди, кластери, аномалії | ClickHouse, OpenSearch | ML‑пайплайн |
| Конвеєри | стадії збір→інсайти | Kafka, Redis, Postgres | моніторинг |
| Моделі | метрики моделей, дрейф | ML‑сховище, Prometheus | реєстр моделей |
| OSINT | скани, результати, статуси | OSINT‑інструменти | SpiderFoot/Amass/etc |
| Звіти | експорт, історія | Postgres, MinIO | PDF/CSV |
| Моніторинг системи | інфра метрики | Prometheus, Loki | Grafana |
| Налаштування | ролі, доступи, інтеграції | Keycloak, БД | автентифікація |

## 3. Архітектура даних
- PostgreSQL: факти, кейси, аудит, якість даних.
- ClickHouse: аналітичні обчислення, тренди.
- Neo4j: графові звʼязки і алгоритми.
- OpenSearch: повнотекстовий пошук.
- Qdrant: векторний пошук.
- MinIO: сирі дані та документи.
- Kafka/Redis: інгестія + статуси.

## 4. UX і функції
- Розслідування: кейси, докази, таймлайн, граф.
- Хронологія: події, фільтри, порівняння сутностей.
- Граф: взаємозвʼязки, підграфи, впливи.
- ГІС: Mapbox або Leaflet, теплові карти.
- Копілот: пояснення аномалій, стислий підсумок.
- Конструктор історій: наративи, знімки, звіти.

## 5. Реалтайм
- SSE/WS для потоків подій, сповіщень, статусів.
- Обовʼязкове повторне зʼєднання і узгодженість стану.

## 6. Безпека і ролі
- Keycloak OIDC.
- Ролі: Аналітик, Слідчий, Адміністратор, Комплаєнс.
- Маскування чутливих даних.
- RLS + WORM для критичних таблиць.

## 7. Продуктивність
- Lighthouse ≥ 90.
- Затримка API < 200ms.
- Час відмальовування панелі < 2s.
- Пагінація, віртуалізація, ліниве завантаження.

## 8. Спостережуваність
- Prometheus + Grafana + Loki.
- Метрики SLO: затримка, рівень помилок, доступність.
- Alertmanager для критичних інцидентів.

## 9. Тестова матриця

| Категорія | Інструмент | Ціль |
|---|---|---|
| Модульні | Pytest/Vitest | 0 критичних падінь |
| Інтеграційні | Pytest + БД/Kafka/Neo4j | узгодженість даних |
| Е2Е | Playwright | основні флоу без помилок |
| Продуктивність | k6/Lighthouse | затримка/оцінка ≥ ціль |
| Безпека | ruff/mypy + сканування | 0 критичних CVE |

## 10. Критерії приймання
- Усі модулі присутні та працюють з реальними даними.
- Всі тести успішні.
- Граф, таймлайн, OSINT працюють з реальних джерел.
- Повна україномовність інтерфейсу.
- Метрики продуктивності в межах порогів.

## 11. Артефакти
- Звіт аудиту.
- Логи та метрики.
- Звіти тестів.
- Документи розгортання.
