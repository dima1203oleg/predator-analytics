## 2. TORNADO INSIGHTS — БІЗНЕС-ЦІННІСТЬ ТА ГЛИБОКЕ ЗАНУРЕННЯ

Tornado Insights — це когнітивний радар, який сканує шум ринку та виділяє з нього критичні сигнали, трансформуючи хаос сирих даних у чіткі проактивні бізнес-рішення. Назва відображає здатність системи вловлювати початкові, ледь помітні «завихрення» ринку, які передують потужним змінам.

### 2.1. Прогнозування попиту та товарних потоків
· Джерела: митна статистика (MD Declaration), податкові накладні, логістичні треки (AIS, Railway API), макроекономічні індикатори.
· Моделі: Ансамбль з Temporal Fusion Transformer (TFT), LightGBM (для табличних даних), Prophet (сезонна декомпозиція), Bayesian Structural Time Series (BSTS). Верифікація через крос-валідацію на історичних даних.
· Вихід: Прогноз «вистрільних» категорій з MAPE ≤ 14%, оптимальні вікна закупівлі/продажу, цінові коридори, виявлення зон перегріву/просадки.
· Ефект: Зниження неліквіду на 30-40%, зростання маржинальності на 15-25%, скорочення касових розривів.

### 2.2. Глибинний аналіз ринкових структур
· Графовий підхід: Динамічна побудова мультиграфу ринку. Кластеризація спільнот (Louvain, Leiden), аналіз центральності (PageRank, Betweenness), виявлення ключових мостів, шлюзів та структурних дір.
· Ціль: Виявлення картельних змов, транзитних хабів, спільних бенефіціарів, прихованих монополій та тіньових ланцюгів постачання.
· Ефект: Точне стратегічне позиціонування, уникнення конфлікту інтересів, виявлення цілей для M&A.

### 2.3. Due Diligence 2.0 з верифікацією ланцюжків
· Entity Resolution: Гібридний підхід. Fuzzy matching (RapidFuzz) поєднується з графовою канонізацією, cross-referencing з державними реєстрами, судовими базами та тендерними майданчиками.
· Risk Scoring: Вагована модель на основі градієнтного бустінгу (XGBoost) з вхідними фічами: PEP, санкції (OFAC, EU, РНБО), судові рішення, податкові розриви, медіа-негатив.
· Юридична валідність: Генерація звітів з цифровими печатками (PKI), хеш-валідацією (BLAKE3), повним chain-of-custody логом (WORM).
· Ефект: Скорочення часу DD на 70%, зниження ризику токсичних контрактів на 85%.

### 2.4. Виявлення аномалій та шахрайських схем

· Детекція: Розриви постачання, штучні ціни, фіктивні контракти, «віялові» платежі, циклічні транзакції, ознаки відмивання коштів.

## 3. АНАЛІТИЧНИЙ КОНТЕКСТ ТА ГЛИБОКА АРХІТЕКТУРНА ПАРАДИГМА

Платформа базується на гібридній парадигмі Event-Driven + CQRS + Graph-Native + Vector-Hybrid, що забезпечує максимальну гнучкість і продуктивність.

### 3.1. Семантичний шар даних
· Динамічна онтологія: Створена на базі OWL/RDF, онтологія бізнес-сутностей (Товар, Контрагент, Угода, Ризик) з можливістю гарячого розширення без зупинки сервісів.
· Версіонування схем: Повна backward/forward compatibility через JSON Schema Registry. Усі події в шині даних (Redpanda) містять посилання на версію схеми.
· Data Lineage: Побудова криптографічно підписаного ланцюга трансформацій на основі дерева Меркла для кожного інсайту, що дозволяє верифікувати джерело та шлях будь-якого факту.
· Контекстно-залежна маршрутизація: Запити спрямовуються за семантичною вагою, а не лише ключовими словами. "Хто кришує компанію Х" автоматично транслюється в графову аналітику та OSINT-пошук.

### 3.1.1. Онтологія графової бази (Neo4j)

Графова база Neo4j є фундаментом для виявлення прихованих патернів серед 100+ аналітичних датасетів. Онтологія визначає ключові сутності та зв'язки для митних даних.

**Ключові сутності (Nodes):**
- **Company** — Компанія/Юридична особа/ФОП (edrpou, name, status, registration_date, is_fop)
- **Person** — Фізична особа (inn, full_name, citizenship, is_pep, is_sanctioned)
- **Declaration** — Митна декларація (decl_id, date, type, total_invoice_value, currency)
- **Product** — Товар/УКТЗЕД (hs_code, description, category)
- **CustomsPost** — Митний пост/КПП (post_code, name, region, location)
- **Broker** — Митний брокер (license_id, name)
- **Address** — Локація/Юридична адреса (address_hash, full_address)
- **Country** — Країна (iso_code, name, is_offshore, is_sanctioned)

**Типи зв'язків (Relationships):**

*Бізнес-зв'язки:*
- `(Person)-[:DIRECTS {appointment_date}]->(Company)` — Особа керує компанією
- `(Person)-[:OWNS {share_percent}]->(Company)` — Особа володіє часткою компанії
- `(Company)-[:REGISTERED_AT]->(Address)` — Компанія зареєстрована за адресою

*Митні операції:*
- `(Company)-[:FILED]->(Declaration)` — Компанія подала декларацію
- `(Broker)-[:PROCESSED]->(Declaration)` — Брокер оформив декларацію
- `(Declaration)-[:CLEARED_AT]->(CustomsPost)` — Декларація пройшла через митний пост
- `(Declaration)-[:CONTAINS {net_weight, gross_weight, invoice_value, quantity}]->(Product)` — Декларація містить товар
- `(Declaration)-[:ORIGINATES_FROM]->(Country)` — Товар походить з країни
- `(Declaration)-[:DISPATCHED_FROM]->(Country)` — Товар відправлено з країни

**Архітектурні принципи:**
1. **Аналітична вага на ребрах** — Суми, обсяги, вага зберігаються на зв'язках для швидкого проходження графом без підняття важких таблиць.
2. **Унікальність через constraints** — Жорсткі обмеження унікальності запобігають дублюванню даних.
3. **Індекси для аналітики** — Індекси оптимізовані для типових аналітичних запитів (дати, суми, коди).
4. **Гнучкість для 100+ датасетів** — Онтологія підтримує всі типи аналітичних патернів: бізнес-зв'язки, митні операції, географічні залежності.

Детальна документація онтології доступна в `docs/neo4j_ontology.md`.

### 3.2. Нейро-оркестратор (LLM Agentic Reasoning Layer)

Нейро-оркестратор — це інтелектуальний прошарок між AI та базами даних, який перетворює природну мову на складні графи та забезпечує автономний аналіз даних. Він є "нервовою системою" PREDATOR Analytics, що дозволяє системі працювати як автономна зброя фінансової розвідки.

**ReAct (Reasoning + Acting) Loop:**

Агент працює за циклом: Отримати запит → Створити Cypher-запит → Виконати → Побачити помилку або порожню відповідь → Самостійно виправити запит → Видати результат.

**Переваги:**
- **Автономна самокорекція** — Агент самостійно виправляє помилки в запитах
- **Прозорість мислення** — Кожен крок логується для аудиту
- **Адаптивність** — Агент адаптується під різні типи запитів

**Tool Calling API (Функціональні виклики):**

Набір жорстко зашитих функцій (Tools), які агент може викликати для виконання специфічних завдань:

- **search_graph_anomaly** — Пошук аномалій в графі за заданим патерном
- **get_entity_details** — Отримання детальної інформації про сутність (компанію, особу, декларацію)
- **analyze_risk_score** — Аналіз ризик-скорингу компанії або особи

**Переваги:**
- **Модульність** — Кожен інструмент може бути розроблений та тестований ізольовано
- **Безпека** — Інструменти обмежені конкретними операціями
- **Масштабованість** — Легко додавати нові інструменти

**Vector Semantic Router:**

Перед тим як генерувати запит, система через Qdrant шукає в базі "схожі минулі розслідування", щоб зрозуміти контекст і використати вже перевірені алгоритми пошуку.

**Переваги:**
- **Контекстна обізнаність** — Агент використовує досвід попередніх розслідувань
- **Прискорення** — Схожі запити обробляються швидше
- **Навчання** — Система покращується з кожним розслідуванням

**Інтеграція з DeepSeek R1:**

Нейро-оркестратор використовує DeepSeek R1 як основну LLM модель для генерації запитів та аналізу результатів.

Детальна документація доступна в `docs/neuro_orchestrator.md`.

### 3.3. Фізична архітектура (Single Server)
Система розгортається через Docker Compose v2 з використанням profiles для можливості гнучкого запуску тільки необхідних компонентів. Усі внутрішні комунікації відбуваються в ізольованому Docker-нетворку predator-net з шифруванням через mTLS, що забезпечується SPIFFE/SPIRE.

## 4. SM RESOURCE CONTRACT — ЖОРСТКІ ОБМЕЖЕННЯ ІНФРАСТРУКТУРИ

Платформа жорстко оптимізована для роботи на одному виділеному сервері.

| Ресурс | Ліміт | Стратегія оптимізації та алгориmічний захист |
|---|---|---|
| GPU | NVIDIA GTX 1080 (8GB VRAM) | GGUF Q4_K_M, KV cache reuse, speculative decoding, automatic layer offloading to CPU/RAM. |
| CPU | ≤ 8 активних ядер | Повністю асинхронний I/O, Ray/Pipe паралелізм, вивантаження криптографічних та графових операцій у WASM. |
| RAM | ≤ 48 GB | Memory pooling, стрімінг чанків, zero-copy читання з MinIO, mmap для великих векторних індексів. |
| Storage | NVMe 2TB+ | Append-only Parquet/ZSTD для даних, LSM-tree індексація, щоденний компактинг. |

**Алгоритм Graceful Degradation:**
1. VRAM > 7.2GB: Автоматичний fallback на CPU+RAM інференс через llama.cpp без втрати запиту.
2. CPU > 85% (sustained 5s): Активація batch-режиму. Оновлення дашборду знижується з 2s до 10s. Відключаються non-critical постобробки.
3. RAM > 43GB: Запуск LRU eviction для кешу.
4. Захист від OOM: Кожен Celery воркер має жорсткий ліміт пам'яті (max-memory-per-child). Усі процеси обгорнуті в systemd з restart=always та затримкою для уникнення crash-loop'ів.

Google Antigravity — це не просто IDE, а автономний ШІ-кодер, що виступає в ролі Senior Architect. Інтеграція через Agent Development Kit v2.1 створює ієрархічну команду агентів, здатних до самостійного планування та реалізації складних архітектурних змін.

### Ключові принципи роботи ADK:
· Surgical Changes: Атомарні патчі з автоматичними rollback-сценаріями. Агент змінює лише те, що необхідно.
· Mandatory Planning Mode: Симуляція змін до застосування з оцінкою ризиків (risk-scoring) та валідацією графу залежностей. Критичні зміни (схема БД, мережеві політики) вимагають human-in-the-loop approval.
· Progressive Disclosure: Агенту надається лише релевантний контекст через SKILL.md.
· Memory & State: Персистентний контекст, логи викликів інструментів, самокорекція на основі попередніх помилок.

### Процес розробки (CI/CD):
1. Агент створює фічу-бранч та атомарні коміти згідно з Conventional Commits.
2. CI Pipeline: Лінтинг (ruff), статичний аналіз (mypy), безпековий скан (bandit, safety), юніт-тести (pytest).
3. Staging: Canary деплой в ізольоване середовище та прогон інтеграційних/E2E тестів (Playwright, Google Test для C++ модулів).
4. Human Review (Для критичних змін): Створюється Pull Request з авто-згенерованим описом та оцінкою впливу.
5. Production Deploy: Blue-green деплой з інтелектуальним health-check (перевірка не тільки статусу, а й бізнес-метрик).

## 7. FUEL-BASED INTELLIGENCE ARCHITECTURE — РОЗУМНЕ УПРАВЛІННЯ LLM

Концепція Fuel розглядає обчислювальні ресурси як стратегічне паливо, що виділяється на основі пріоритету та очікуваної цінності.

### 7.1. Tier-архітектура LLM
| Tier | Призначення | Модель | VRAM | Use Case |
|---|---|---|---|---|
| Tier-0 (Critical) | Максимальна точність | Ensemble (Qwen2-7B + Phi-3) | ~6.5GB | Due Diligence, Security, Legal |
| Tier-1 (High) | Висока точність та надійність | MoE Router | ~4.2GB | Forecasting, Anomaly Detection |
| Tier-2 (Medium) | Збалансоване рішення | Qwen2-1.5B / Gemma-2B | ~1.8GB | Market Pulse, Summaries |
| Tier-3 (Low) | Швидкість та економія | CPU-Only Tiny Model | 0GB | Logging, Basic Queries |

### 7.2. Динамічне управління «паливом»
· Бюджетування: Квоти на користувача/департамент на день/тиждень.
· Throttling: Пріоритетні черги при пікових навантаженнях.
· Energy-aware: Зниження точності (FP16 -> INT8) при тепловому тротлінгу GPU.
· Auto-fallback: LLM → Rule Engine → Heuristic → Cache Hit → Відмова з поясненням.
· Моніторинг: Глибоке відстеження W/insight (вартість одного інсайту) та tokens/sec.
