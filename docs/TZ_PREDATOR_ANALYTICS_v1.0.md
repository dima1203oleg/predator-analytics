## 1. ЗАГАЛЬНІ ПОЛОЖЕННЯ

### 1.1. Призначення системи
**Predator Analytics** — це інтелектуальна аналітична платформа класу Palantir, призначена для автоматизованого збору, обробки, збагачення та аналізу даних з державних реєстрів України, відкритих джерел (OSINT) та міжнародних баз для виявлення прихованих зв'язків, оцінки ризиків та розслідування схем у сфері логістики, митного контролю, фінансового моніторингу та комплаєнсу.

### 1.2. Цілі створення системи
- Консолідація понад **250+ державних реєстрів України** та **200+ OSINT-інструментів** у єдиному аналітичному просторі.
- Автоматичне виявлення фіктивних імпортерів, компаній-прокладок, схем з митною вартістю та контрабандних маршрутів.
- Побудова графа знань (Knowledge Graph) для візуалізації зв'язків між сутностями (люди, компанії, майно, події).
- Забезпечення Prompt-Guided Analytics (аналітика природною мовою) на основі RAG (Retrieval-Augmented Generation).

### 1.3. Нормативно-правова база
- Закони України "Про захист персональних даних", "Про доступ до публічної інформації", "Про відкриті дані".
- Вимоги Національного центру резервування щодо роботи з реєстрами.
- Міжнародні стандарти STIX 2.1 для обміну інформацією про загрози.

## 4. ГРАФОВА МОДЕЛЬ (KNOWLEDGE GRAPH)

### 4.1. Онтологія за стандартом STIX 2.1

Система базується на розширеній онтології STIX 2.1 для characterization of entities та виявлення прихованих зв'язків.

#### 4.1.1. Типи вузлів (Nodes)

| Тип сутності | Атрибути | Приклад |
|--------------|----------|---------|
| `Threat-Actor` | ПІБ, РНОКПП, дата народження, паспорт | Фізична особа |
| `Organization` | ЄДРПОУ, назва, статус, КВЕД, адреса | Юридична особа, ФОП |
| `Infrastructure` | IP, домен, ASN, хостинг-провайдер | Сервери, мережеве обладнання |
| `Indicator` | email, телефон, VIN, кадастровий номер | Індикатори |
| `Location` | координати, адреса, регіон, країна | Геолокація |
| `Document` | судове рішення, тендерна документація | Файли, тексти |
| `Event` | дата, тип події, учасники | Судові засідання, реєстрація |

#### 4.1.2. Типи зв'язків (Edges)

| Edge | Джерело → Ціль | Семантика |
|------|----------------|-----------|
| `owns` | Person → Organization | Засновник/власник |
| `manages` | Person → Organization | Керівник |
| `registered_at` | Organization → Location | Юридична адреса |
| `involved_in` | Organization → Event | Сторона у справі |
| `has_indicator` | Threat-Actor → Indicator | Використовує email/телефон |
| `resolves_to` | Domain → IP | Домен вказує на IP |
| `connected_with` | Person → Person | Спільна діяльність |

### 4.2. NLP-пайплайн для наповнення графа

Для автоматичного видобування сутностей з неструктурованих текстів (судові рішення, новини):

1. **Entity Extraction (NER)**: Використання навчених моделей (spaCy, Hugging Face) для ідентифікації ПІБ, компаній, адрес, номерів справ.
2. **Coreference Resolution**: Визначення, що "Петров", "він", "директор ТОВ "Х" — одна сутність.
3. **Relationship Extraction**: Автоматичне визначення зв'язків між сутностями з тексту.
4. **Sentiment Analysis**: Оцінка тональності згадок (позитивний/негативний контекст).

### 4.3. Prompt-Guided Analytics (RAG + Graph)

Архітектура для аналізу природною мовою:

- **LangChain** + **Qdrant** (векторна БД) для семантичного пошуку.
- **TypeDB MCP Server** для перетворення природномовних запитів у запити до графової БД.
- **LLM** (Claude, GPT) для формування відповідей на основі отриманого підграфа.

**Приклад роботи**:
```
Запит: "Покажи всі компанії, пов'язані з Івановим Петром, 
        які мають податкові борги і вигравали тендери ДП "Укрспирт""

Результат: Граф із зв'язками + текстове пояснення.
```

## 5. АНАЛІТИЧНІ МОДУЛІ

### 5.1. AML Scoring (Оцінка ризиків)

**Фактори ризику** (вагова модель):

| Фактор | Вага |
|--------|------|
| Наявність у санкційних списках | 100 |
| Участь у судових справах про контрабанду | 80 |
| Податкові борги > 1 млн грн | 70 |
| Зв'язки з офшорними юрисдикціями | 60 |
| Часта зміна керівників/засновників | 40 |
| Відсутність реальної діяльності (нульова звітність) | 50 |
| Масова реєстрація на одну адресу | 30 |

**Результат**: Ризик-скор (0-100) для кожної сутності.

### 5.2. Anomaly Detection (Виявлення аномалій)

- **Тимчасові ряди**: Аналіз динаміки імпорту/експорту за кодами УКТЗЕД, виявлення різких змін (методи Фур'є, вейвлет-аналіз).
- **Кластерний аналіз мереж**: Виявлення груп компаній зі спільними власниками/IP/телефонами.
- **Паттерн-матчинг**: Пошук відомих схем (наприклад, "скрутка" ПДВ, "пересорт" товарів).

### 5.3. Link Analysis (Аналіз зв'язків)

- **Degree Centrality**: Виявлення центральних фігур (хаби).
- **Betweenness Centrality**: Виявлення "мостів" між групами.
- **Коефіцієнт кластеризації**: Виявлення щільних груп.
- **Найкоротші шляхи**: Пошук зв'язку між двома сутностями.
- **Community Detection**: Автоматичне виділення спільнот (Louvain, Label Propagation).

## 7. ВИМОГИ ДО БЕЗПЕКИ

### 7.1. Рольова модель доступу (RBAC)

| Роль | Права |
|------|-------|
| **admin** | Повний доступ, керування користувачами, джерелами |
| **analyst** | Пошук, перегляд графів, створення звітів, AI Copilot |
| **operator** | Завантаження даних, моніторинг інгестії |
| **viewer** | Тільки перегляд (read-only) |

### 7.2. Аудит

- **Журналювання всіх дій**: хто, коли, що шукав, які дані переглядав.
- **WORM-таблиці**: Неможливість видалення/зміни логів аудиту.
- **Retention policy**: Зберігання логів мінімум 5 років.

### 7.3. Захист даних

- Шифрування даних у спокої (AES-256).
- TLS 1.3 для передачі даних.
- Ізоляція даних різних клієнтів (multi-tenancy з RLS).
- Секрети тільки через environment variables (ніколи в коді).
