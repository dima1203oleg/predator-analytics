## Том 1 — Vision & Strategy

розвідки з відкритих джерел, що автоматично інтегрує, нормалізує,
зв'язує та пояснює інформацію для прийняття обґрунтованих рішень.

аналітичну роботу сотень фахівців автономними AI-агентами, забезпечуючи
швидкість, точність, пояснюваність та масштабованість.

### 3.2 Ontology Governance

- Кожна онтологія має Custodian Agent (відповідального AI-агента)
- Зміни в Core Ontology проходять через Approval Gate
- Версіонування через Semantic Versioning
- Автоматична валідація через SHACL Shapes
- Backward Compatibility або керована міграція

## Том 4 — Universal Knowledge Graph

### 4.1 Graph Layers

Knowledge Graph складається з незалежних шарів, кожен з яких має власну
схему, правила індексації та політики доступу:

| Шар | Призначення |
|---|---|
| **Entity Graph** | Усі сутності: компанії, особи, активи |
| **Event Graph** | Усі події з часовими мітками |
| **Evidence Graph** | Доказова база з ланцюгами верифікації |
| **Document Graph** | Документи, їх версії та зв'язки |
| **Timeline Graph** | Хронологічні послідовності подій |
| **Communication Graph** | Комунікації між сутностями |
| **Corporate Graph** | Корпоративні структури та ієрархії |
| **Trade Graph** | Торговельні потоки та ланцюги |
| **Financial Graph** | Фінансові потоки та транзакції |
| **Ownership Graph** | Структури власності |
| **Beneficial Ownership Graph** | Бенефіціарні власники |
| **Family Graph** | Родинні та особисті зв'язки |
| **Network Graph** | Мережі впливу та взаємодії |
| **Influence Graph** | Політичний та бізнесовий вплив |
| **Asset Graph** | Активи, нерухомість, транспорт |
| **Geo Graph** | Географічні зв'язки та маршрути |
| **Risk Graph** | Ризик-фактори та їх взаємозв'язки |
| **Sanctions Graph** | Санкційні списки та зв'язки |
| **Investigation Graph** | Справи, гіпотези, висновки |
| **AI Memory Graph** | Знання, набуті AI в процесі роботи |

### 4.2 Knowledge Processing Pipeline

```
Ingestion → Extraction → Resolution → Normalization →
Deduplication → Enrichment → Verification → Scoring →
Linking → Indexing → Knowledge Graph → Explainable AI
```

Усі дані автоматично: нормалізуються, дедуплікуються, збагачуються,
верифікуються, оцінюються, зв'язуються, індексуються, потрапляють до
Knowledge Graph і стають доступними для Explainable AI.

- Конфліктні записи вирішує Arbiter Agent

## Том 12 — Enterprise Security Constitution

### 12.1 Security Principles

| Принцип | Визначення |
|---|---|
| **Zero Trust** | Ніхто не довіряється за замовчуванням |
| **Defense in Depth** | Багаторівневий захист |
| **Least Privilege** | Мінімально необхідні права |
| **Need to Know** | Доступ тільки до необхідної інформації |
| **Need to Use** | Доступ тільки до необхідних функцій |
| **Secure by Default** | Безпечна конфігурація за замовчуванням |
| **Privacy by Default** | Мінімум даних за замовчуванням |
| **Privacy by Design** | Конфіденційність як архітектурний принцип |
| **Security by Design** | Безпека як архітектурний принцип |
| **Immutable Audit** | Незмінні журнали аудиту (WORM) |
| **Continuous Verification** | Постійна перевірка стану безпеки |

### 12.2 Security Toolchain

| Інструмент | Призначення |
|---|---|
| **Policy as Code** | OPA / Kyverno |
| **Secrets Management** | HashiCorp Vault / Sealed Secrets |
| **Supply Chain Security** | SBOM, SLSA, Sigstore, Cosign |
| **Runtime Security** | Falco |
| **Vulnerability Scanning** | Trivy |
| **Threat Intelligence** | Інтегроване через Connector Ecosystem |

### 12.3 Security Hard Rules

```
SEC-01  Секрети НІКОЛИ в коді — тільки env vars або Vault
SEC-02  Docker ЗАВЖДИ multi-stage, НІКОЛИ root
SEC-03  Pod ЗАВЖДИ resource limits (cpu + memory)
SEC-04  Мережеві політики: deny-all за замовчуванням
SEC-05  mTLS для всіх внутрішніх комунікацій
SEC-06  Audit log: immutable, append-only (WORM)
SEC-07  Залежності без оновлень > 1 року: ЗАБОРОНЕНО
SEC-08  Зовнішні SaaS (Sentry, GA тощо): ЗАБОРОНЕНО
SEC-09  Кожен контейнер: read-only root filesystem
SEC-10  Усі API: rate limiting + input validation
```

```
INV-12  SELECT *: ЗАБОРОНЕНО

INV-14  Pod без resource limits: ЗАБОРОНЕНО
INV-15  Зовнішні SaaS: ЗАБОРОНЕНО
INV-16  WORM таблиці: UPDATE/DELETE = ERROR
INV-17  MacBook = IDE ONLY, НІКОЛИ compute node
```

### 20.2 Обов'язкові Стандарти

| Область | Стандарт |
|---|---|
| Python | 3.12 ONLY, Mypy strict, Ruff |
| TypeScript | Strict mode, ESLint |
| API | OpenAPI 3.1, AsyncAPI 3.0 |
| Events | CloudEvents 1.0 |
| Commits | Conventional Commits, українською |
| Docs | Markdown, Mermaid, українською |
| Tests | ≥ 85% coverage |
| Security | OWASP ASVS Level 2 |
| Containers | Multi-stage, non-root, read-only fs |

### 20.3 Заборонені Практики

- Hardcoded credentials або секрети
- `Any` без `# type: ignore` з коментарем
- `SELECT *` замість конкретних колонок
- Англійська в UI (= критична помилка HR-04)
- Залежності без оновлень > 1 року
- Прямий доступ до БД з UI (тільки через API)
- Мутація стану без Event (Event Sourcing)
- Монолітний фронтенд (тільки component architecture)
- Deploy без проходження CI pipeline

### 20.4 Критерії Прийняття Рішень

Пріоритет при неоднозначності (від вищого до нижчого):

1. **Безпека** та дотримання законодавства
2. **Якість архітектури** та AI Constitution
3. **Підтримуваність** та читабельність коду
4. **Масштабованість** та продуктивність
5. **Довгострокова еволюція** платформи
6. **Швидкість** реалізації

### 20.5 Політики Еволюції

- Backward Compatibility обов'язкова для public API
- Deprecation: мінімум 2 мінорні версії перед видаленням
- Breaking Changes: тільки в major версіях
- Migration Guides: обов'язкові для кожного breaking change
- Feature Flags: для поступового впровадження
