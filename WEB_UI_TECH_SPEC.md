# ФІНАЛЬНЕ ТЕХНІЧНЕ ЗАВДАННЯ (ТЗ)

## Покращення веб-інтерфейсу Predator Analytics
**(Stability-first, UX-driven, Non-destructive)**

---

### 1️⃣ Мета

Вдосконалити існуючий веб-інтерфейс Predator Analytics, зробивши його:
*   більш інтуїтивним
*   більш візуально зрозумілим
*   більш продуктивним
*   більш передбачуваним у поведінці

**❗ Ключова умова:**
Усі покращення мають бути **НЕруйнівними** — існуючі розділи, підрозділи, логіка та API не видаляються і не перезаписуються.

---

### 2️⃣ Обовʼязкові принципи роботи

#### 🔒 Non-destructive rule
*   ❌ НЕ видаляти існуючі сторінки
*   ❌ НЕ ламати поточну навігацію
*   ❌ НЕ міняти API без сумісності
*   ❌ НЕ “переробляти дизайн з нуля”

#### ✅ Дозволено
*   Покращувати, доповнювати, оптимізувати
*   Додавати нові елементи поруч, а не замість
*   Робити UI більш зрозумілим без зміни сенсу

---

### 3️⃣ Загальні вимоги до UX / UI

#### 3.1 Інтуїтивність
Інтерфейс повинен:
*   чітко показувати що зараз відбувається
*   не мати “магічних” дій без пояснення
*   мати явні стани: `loading`, `empty`, `success`, `error`

#### 3.2 Візабіліті
Потрібно:
*   підвищити читабельність
*   структурувати інформацію
*   зменшити когнітивне навантаження

Дозволено: підсвітки, статусні бейджі, іконки, таймлайни, прогрес-індикатори.

---

### 4️⃣ Структура інтерфейсу (зберігається)

Усі існуючі розділи ЗАЛИШАЮТЬСЯ.
Мінімальні логічні зони, які мають бути зрозумілими:

1.  **Dashboard** (стан системи, активність, помилки)
2.  **Documents / Data** (статуси, прогрес, tenant-контекст)
3.  **Ingestion / Jobs** (історія, черги, помилки)
4.  **Search / Analysis** (завжди зрозуміло, що шукається)
5.  **System / Health** (version, environment, metrics)

❗ *Якщо розділ складний — дозволено додати пояснювальний UI, але не прибирати функціонал.*

---

### 5️⃣ Додавання нових фіч (обережно)

Дозволяється додавати:
*   інтерактивні дашборди
*   фільтри
*   збережені пресети
*   кастомізацію віджетів
*   quick actions
*   contextual hints (пояснення кнопок)

❗ **Усі нові фічі мають бути опціональними та не повинні змінювати поведінку існуючих.**

---

### 6️⃣ Узгодженість UI ↔ API

*   UI повинен працювати з одним канонічним API.
*   ❌ НЕ змішувати `/api/v1` і `/api/v25`.
*   Кожен запит: `tenant-aware`, передбачуваний.
*   Помилки API повинні відображатись коректно та бути зрозумілими користувачу.

---

### 7️⃣ Оптимізація продуктивності

Потрібно:
*   зменшити зайві запити
*   усунути дубль fetch
*   мінімізувати непотрібні ререндери
*   використовувати кешування там, де можливо

❗ *Оптимізація без зміни логіки роботи.*

---

### 8️⃣ Поведінка при змінах

Будь-яка зміна:
*   має бути сумісна назад
*   не повинна “ламати” старі сценарії
*   має супроводжуватись коротким поясненням (коментар / README)

---

### 9️⃣ Очікуваний результат

Після виконання ТЗ:
*   інтерфейс виглядає як єдиний продукт
*   користувачу зрозуміло: що відбувається, де він знаходиться, який стан системи
*   нічого не пропало і нічого не зламалось
*   зʼявилась чіткість і впевненість

---

### 🔟 Головний принцип

**Покращуй, але не руйнуй. Оптимізуй, але не стирай. Пояснюй, а не ховай.**
